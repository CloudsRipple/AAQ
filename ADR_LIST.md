# ADR 列表（To-Be）

更新时间：2026-03-15  
状态约定：`Accepted / Proposed / Deprecated`

## ADR-001：采用六层领域分层并强制单向依赖
- 状态：Accepted
- 背景：当前编排层职责过重，策略、风控、执行、数据加载耦合在同一编排文件中。
- 决策：采用 `market_data -> signal_engine -> risk_engine -> execution_engine -> state_store -> observability` 单向分层。
- 结果：降低耦合与回归范围；代价是需要引入接口与契约适配层。

## ADR-002：订单提交采用稳定幂等键（intent_id）
- 状态：Accepted
- 背景：当前去重依赖内存，重启后可能重复提交。
- 决策：`intent_id` 由稳定字段生成（策略事件+风险决策+版本），并在 state_store 建立唯一约束。
- 结果：可防重复下单；代价是执行层需维护幂等冲突处理路径。

## ADR-003：系统必须“先对账后交易”
- 状态：Accepted
- 背景：重启/断连后本地状态与券商状态可能错位。
- 决策：引入系统状态机 `BOOTSTRAP -> RECONCILE -> RUNNING`，未完成对账不得进入 RUNNING。
- 结果：提升恢复一致性；代价是启动时间增加、需要对账实现。

## ADR-004：风险引擎不可用时执行 fail-closed（仅减仓）
- 状态：Accepted
- 背景：风险引擎不可用属于资金风险事件，不能继续新开仓。
- 决策：风险引擎异常进入 `DEGRADED`，阻断新开仓，仅允许保护性减仓；持续异常转 `HALTED`。
- 结果：降低尾部风险；代价是部分行情机会放弃。

## ADR-005：禁止生产环境使用“默认快照”作为交易数据
- 状态：Accepted
- 背景：数据源失败时回退硬编码快照会产生伪信号交易。
- 决策：生产模式下数据断流触发降级/停机，不允许使用默认快照开新仓。
- 结果：避免脏数据交易；代价是在数据故障时交易可用性下降。

## ADR-006：执行回报采用双状态记录（broker_status + local_status）
- 状态：Accepted
- 背景：单一状态字段无法表达“本地已提交但券商未确认”等错位场景。
- 决策：`ExecutionReport` 同时记录券商状态与本地状态，并要求状态转移事件可追溯。
- 结果：便于定位状态错位；代价是状态机与审计复杂度上升。

## ADR-007：配置分层治理与生产安全默认
- 状态：Accepted
- 背景：当前配置默认值偏开发友好，生产安全边界不清晰。
- 决策：配置分 `static/runtime/secret` 三类；生产环境默认 fail-safe，危险回退需显式开关并审计。
- 结果：降低误配置风险；代价是配置管理流程更严格。
