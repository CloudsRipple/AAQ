# 分阶段落地清单（最小可行改造路径）

更新时间：2026-03-15  
原则：小步可回滚、先止血后扩展、每阶段必须可验收。

## 阶段0：契约冻结与防线前置
- 目标：
  - 冻结四类核心事件契约：`SignalEvent / RiskDecision / OrderIntent / ExecutionReport`
  - 冻结订单状态机与系统状态机
  - 建立契约兼容桥接（旧结构到新契约映射）
- 工作量：中
- 主要风险：契约字段定义不完整导致后续返工
- 验收标准：
  - 契约文档评审通过并版本化
  - 现有主链可通过桥接层产出完整 `trace_id` 链路
  - 不改业务行为的回归测试全通过

## 阶段1：执行幂等与状态持久化最小闭环
- 目标：
  - 引入 `intent_id` 唯一约束与幂等索引
  - 增加订单账本（意图、提交、回报、转移）与持仓快照表
  - 实现“同 intent 重放不重复发单”
- 工作量：高
- 主要风险：写路径改造影响执行稳定性
- 验收标准：
  - 重放同一 `intent_id` 不产生重复下单
  - 重启后可恢复未完成订单状态
  - 故障注入（进程崩溃后恢复）场景通过

## 阶段2：启动对账与运行状态机落地
- 目标：
  - 落地系统状态机 `BOOTSTRAP/RECONCILE/RUNNING/DEGRADED/HALTED`
  - 启动流程强制“先对账后交易”
  - 对账失败自动降级/停机
- 工作量：高
- 主要风险：对账规则不完善导致误停机
- 验收标准：
  - 启动时完成 open orders / positions / trades 对账
  - 对账失败时不会进入 RUNNING
  - 对账修复后可恢复 RUNNING

## 阶段3：降级策略与故障编排
- 目标：
  - 数据断流、网关断连、DB锁冲突、风控不可用四类降级策略落地
  - 实施“风险不可用仅减仓”策略
  - 引入统一重试/退避/熔断配置
- 工作量：中高
- 主要风险：降级路径复杂导致边界行为遗漏
- 验收标准：
  - 四类故障注入场景均有确定状态转移
  - 降级状态下新开仓被阻断
  - 恢复后需先经过 RECONCILE 再恢复 RUNNING

## 阶段4：观测与审计闭环
- 目标：
  - 统一结构化日志与指标（下单成功率、拒单率、延迟、滑点、回撤）
  - 告警与追踪基于 `trace_id/intent_id` 闭环
  - 审计报告自动生成（按交易日）
- 工作量：中
- 主要风险：指标定义不一致导致监控噪声
- 验收标准：
  - 任一成交可回溯至完整链路事件
  - P0/P1 事件可自动告警并携带定位信息
  - 日报可复盘关键决策与执行偏差

## 阶段5：清理遗留与架构收敛
- 目标：
  - 下线临时桥接与重复路径
  - 清理跨层调用与隐式共享状态
  - 固化模块边界检查（静态校验）
- 工作量：中
- 主要风险：清理阶段误删兼容路径
- 验收标准：
  - 核心路径仅保留 To-Be 分层实现
  - 跨层依赖检查通过
  - 关键回归场景全绿

## 跨阶段硬门禁（每阶段都必须满足）
- 不允许引入“无审计发单”路径。
- 不允许绕过 risk_engine 直接进入 execution_engine。
- 不允许生产模式使用默认快照作为交易依据。
- 任一阶段失败必须可一键回滚到上阶段稳定版本。
